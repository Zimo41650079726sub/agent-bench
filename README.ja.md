# agent-bench

[English](README.md) | 日本語

**あなたのエージェントスキルを最後まで正しくやり遂げられる、いちばん小さなローカルLLMを見つけるためのベンチマークです。**

## これは何？

最近のLLMは、ただ文章を書くだけでなく、「ファイルを読む→コマンドを実行する→結果を見て直す」という多段階の作業（エージェント作業）をこなします。ところが、小さなモデルにこれをやらせると、途中でツールの使い方を間違えたり、作業を投げ出したりすることがよくあります。

agent-bench は「このモデルは、この作業手順を最後まで正しくやり切れるのか？」を測るツールです。特徴は採点方法にあります。

- **採点にLLMを使いません。** コマンドの成否（exit code）、ファイルがいつ作られたか、ファイル内容のハッシュ値（SHA256）だけで採点します。だから同じ条件なら誰がいつ測っても同じ結果になります。
- **結果だけでなく過程も見ます。** 測りたい能力を「旗」に分解し、オリエンテーリングのように「どの旗に、何ターン目に届いたか」を記録します。たとえば「バグを直す前に、まずテストを実行して失敗を自分の目で確かめたか？」という作業のお作法も旗の一つです。
- **完全にローカルで動きます。** 手元のPCで動くOpenAI互換サーバー（LM Studio・Ollama・llama-server など）につなぐだけです。

同じ試行を k 回くり返して全部成功したかを `pass^k` と書きます。1回まぐれで成功しただけでは合格になりません。

## 必要なもの

- Python 3.12以上（追加のpip installは不要。標準ライブラリだけで動きます）
- Docker（モデルの作業場所として、使い捨ての隔離環境を作るのに使います）

## 動かし方

```bash
docker build -t agent-bench:latest .
python3 tests/verify_harness.py   # LLMなしの自己診断（モックで採点系をテスト）
python3 cli.py --model YOUR_MODEL --task debug_python_v1 --k 3 \
    --base-url http://localhost:8080/v1
```

自作のスキルを測りたいときは、スキルのフォルダを指定します:

```bash
python3 cli.py --model YOUR_MODEL --task skill_run_v1 \
    --skill-dir ./examples/skills/report-skill --k 5 \
    --base-url http://localhost:8080/v1
```

## ダッシュボード（ブラウザで結果を見る）

```bash
python3 webui.py          # -> http://127.0.0.1:8765/
```

- **実行パネル**: モデルとタスクを選んでボタンを押すと、進行中のやり取りがリアルタイムで流れます。
- **マトリクス表示**: モデル×タスクの成績一覧。
- **行をクリックすると全記録**: モデルがどのツールをどう呼んだか、何を言ったか、全部読めます。
- **成果物プレビュー**: モデルが作ったHTMLや画像を、ブラウザ上で並べて見比べられます。「どのモデルがまともなWebページを作れるか」が一目で分かります。

### 面白いのは「失敗の仕方」

このベンチで一番の発見は、**モデルの能力は大きさの順に並ばない**ということです。しかも各モデルの失敗はまぐれではなく、何度やっても同じ場所で同じように転びます。

- **gemma-4-e2b（たった2B）** が6タスク中5つに合格。兄貴分たちが転ぶ2つの課題（e4bが投げ出すランディングページ、12bが崩壊するダッシュボード）を両方クリアし、合計時間も全モデル中最速（8.7分）。唯一の弱点はTDD（テスト駆動開発）のお作法で、テストを書いた後、それが失敗することを確認せずにいきなり実装を始めてしまう。
- **gemma-4-12b** は5/6合格だが、ダッシュボード課題で崩壊。書き込むHTMLが大きくなると、ファイル書き込みツールの「保存先パス」を指定し忘れるようになる。「エラーの原因はパス指定漏れだ」と自分で正しく言い当てた直後にも、9回連続で同じミスをした。同じ課題を、ずっと小さいe4bは3ターンで済ませている。
- **gemma-4-e4b** は一番重いダッシュボード課題をクリアするのに、それより簡単なランディングページ課題をターン1で放棄する。実在しないファイル名を思い込みで読みに行き、失敗すると「ドキュメントがないので進められません」と諦める。3回やって3回とも同じだった。TDD課題では、テストファイルの中に答えのニセモノ（スタブ）を書いてしまい、自分で作った罠から抜け出せなかった。
- **qwen3.5-4b** がベンチ全体で落とした旗はたった1つ。「コードを実行して失敗を自分の目で見る」という手順を飛ばして、いきなり修正する癖がある。10回測って10回とも同じだったので、これはブレではなく、このモデルの安定した癖。

## 結果ファイルの読み方

結果は `results/result_{タスクID}_{時刻}.json` に保存されます。主な項目:

- `pass_all_k` — k回の試行が全部合格したか
- `skills.{旗の名前}.reach_rate` — 旗ごとの到達率。モデルの癖の診断に使う
- `invalid_tool_call_count` — 存在しないツールを呼ぼうとした回数（小さいモデルの典型的な失敗）
- `tamper_detected_count` — ズル（触ってはいけないファイルの改変）で無効になった試行数
- `trials[]` — 全試行の完全なログ
- `environment` — 実行条件の記録（temperature などのサンプリング設定・サーバー・モデル名）。temperature=0 にしてもローカルサーバーの出力は毎回同じとは限らないので、条件を記録した上で「pass^k のブレ具合」自体を安定性の指標として扱います
- `result_hash` — 結果ファイルの破損・重複チェック用（偽造防止の署名ではありません）

## ズル対策

モデルは思わぬ方法でテストをすり抜けようとすることがあります。agent-bench は次の対策を入れています。

- **作業フォルダ全体の変更チェック**: 許可されたファイル以外を1つでも変更したら（こっそり `conftest.py` を追加する等も含めて）、その試行は無効。
- **モデルの自己申告を信用しない**: `pytest || true` のように「成功したフリ」をするコマンドは通用しません。採点時にハーネスが自分でテストを実行し直します。
- **ファイルの作成順はスナップショットで判定**: ツールを使わずシェルの `echo > file` で書いても、ちゃんと同じように記録されます。

## タスクの追加

`agent_bench/tasks/` に `BenchTask` のサブクラスを1ファイル作り、`setup`（作業場所の準備）/ `get_prompt`（モデルへの指示文）/ `skills`（旗の定義）/ `allowed_writes`（変更を許すファイル）を書いて `tasks/__init__.py` に登録するだけです。実行ループや採点基盤には触れません。

## 自作スキルの測り方

スキルの `SKILL.md` と同じ場所に `bench_manifest.json` を置き、次を宣言します:

- `required_reads` — 読むべきファイル
- `required_commands` — 実行すべきコマンド（正規表現）
- `expected_artifacts` — 完成しているべき成果物
- `allowed_writes` — 変更してよいファイル
- `stubs` — 重い処理（画像生成など）を軽いダミーに差し替える仕組み

このベンチが測るのは「モデルが手順を正しく運転できるか」であって、成果物の芸術的な出来ばえではありません。書き方は `examples/skills/report-skill/` が見本です。どのスキルでも `no_tool_hallucination`（ツールのでっち上げをしない）と `step_completion`（手順を最後までやり切る）の2つの旗は自動で計測されます。
